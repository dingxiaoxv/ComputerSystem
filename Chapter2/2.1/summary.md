## 核心概念

### 十六进制（§2.1.1）

位模式用十六进制表达的原因：可读性高，且和二进制换算简单（每 4 位 = 1 个十六进制位）。

**A-F 对应十进制（需牢记）：**

| 十六进制 | 十进制 | 二进制 |
|---------|--------|--------|
| A | 10 | 1010 |
| B | 11 | 1011 |
| C | 12 | 1100 |
| D | 13 | 1101 |
| E | 14 | 1110 |
| F | 15 | 1111 |

**2 的幂次转十六进制窍门**：若 `x = 2^n`，令 `n = i + 4j`（`0 ≤ i ≤ 3`），则结果为 `2^i` 后跟 `j` 个 `0`。

```
2^11 = 2^(3 + 4×2)  →  i=3, j=2  →  0x800
2^9  = 2^(1 + 4×2)  →  i=1, j=2  →  0x200
2^16 = 2^(0 + 4×4)  →  i=0, j=4  →  0x10000
```

---

### 数据大小（§2.1.2）

不同类型的字节数由编译器和平台决定，**最关键的差异是指针**：

| 类型 | 32 位系统 | 64 位系统（Linux） |
|------|---------|---------|
| `char` | 1 | 1 |
| `int` | 4 | 4 |
| `long` | 4 | **8** |
| 指针 | **4** | **8** |

> `long` 在 Windows 64 位仍为 4 字节，跨平台代码用 `int32_t/int64_t`（`<cstdint>`）。

可移植代码必须用 `sizeof` 而非写死数字。

---

### 字节序（§2.1.3）

多字节数在内存中的排列方式：

- **小端（Little-Endian）**：低有效字节在低地址
  - `0x12345678` 存为 `78 56 34 12`
  - 适用：x86/x86-64、ARM（默认）、Apple Silicon（M 系列）、RISC-V
- **大端（Big-Endian）**：高有效字节在低地址
  - `0x12345678` 存为 `12 34 56 78`
  - 适用：老 SPARC/PowerPC、**网络协议（TCP/IP 强制大端）**

**暴露场景**：字节序只在跨边界时才成问题——网络传输、读写二进制文件、用调试器直接看内存。同一台机器上的纯计算不受影响。

```c
// 网络编程必须显式转换
uint32_t htonl(uint32_t hostlong);  // host → network（大端）
uint32_t ntohl(uint32_t netlong);   // network → host
```

---

### 位运算（§2.1.4-2.1.9）

**掩码运算**：用 `&` 把不关心的位清零，只保留目标位。

```c
x & 0xFF          // 提取低 8 位
x & 0x80000000    // 检查符号位（32 位）
ip & 0xFFFFFF00   // 提取 IPv4 网络地址
```

**移位运算**：

| 类型 | 行为 | 用于 |
|------|------|------|
| 左移 `<<` | 右侧补 0 | 等效乘以 2^k |
| 逻辑右移 `>>` | 左侧补 **0** | 无符号数除以 2^k |
| 算术右移 `>>` | 左侧补**符号位** | 有符号数除以 2^k（向下取整） |

C 标准对有符号数右移未规定行为，但 gcc/clang 均做算术右移；可移植代码不应依赖此行为。

---

## 易错点

1. **有符号数右移是实现定义的**：C 标准不保证，只是 gcc/clang 碰巧都做算术右移。
2. **字节序只在跨边界时暴露**：纯计算感知不到字节序，调试器看内存/网络传输才需要关注。
3. **`long` 大小依赖平台**：Linux 64 位是 8 字节，Windows 64 位是 4 字节。

## 实验记录

### show_bytes —— 验证字节布局

**代码**：`Chapter2/2.1/show_bytes.cpp`

```bash
g++ -std=c++20 -Wall -o show_bytes show_bytes.cpp && ./show_bytes
```

**输出**：
```
int    12345: 39 30 00 00
float  12345: 00 e4 40 46
pointer &i  : 98 10 2e 78 ff 7f 00 00
```

**结论**：
- `0x3039 = 12345`，低字节 `39` 在低地址 → 确认小端序
- float 与 int 位表示完全不同 → 类型决定解释方式，不是数据本身
- 指针 8 字节，高位补零 → x86-64 实际只用 48 位有效地址

---

## 三句话总结

- **最重要的概念**：信息只是字节，类型决定解释方式——同样的位，作为 `int` 和 `float` 读出来完全不同。
- **最容易错的点**：有符号数右移是实现定义行为；`long` 大小依赖平台。
- **对应的工程现象**：`show_bytes` 揭示小端字节序和 IEEE 754 浮点布局；网络编程中 `htonl/ntohl` 正是为了处理字节序差异。
