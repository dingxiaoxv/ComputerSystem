# §2.4 浮点数 总结

## 核心主线

浮点数是实数的**近似**表示，不是精确表示。两个看起来等价的计算顺序可能给出不同结果。IEEE 754 是现代浮点运算的统一标准。

---

## 知识点

### 1. IEEE 754 格式（§2.4.2）

| 类型 | 符号位 | 阶码 | 尾数 | 有效十进制位数 |
|------|--------|------|------|----------------|
| `float`（32位）| 1 | 8 | 23 | ~7 位 |
| `double`（64位）| 1 | 11 | 52 | ~15 位 |

**实际值**：$(-1)^s \times M \times 2^E$

| 阶码值 | 含义 |
|--------|------|
| 全 0 | 非规格化数（含 ±0） |
| 全 1 | 特殊值（±∞、NaN） |
| 其他 | 规格化数，$E$ = 阶码 - 偏置值（float 偏置 127，double 偏置 1023）|

规格化数的尾数 M 隐含前导 1，即 $M = 1.xxx\ldots$，免费多一位精度。

### 2. 舍入规则（§2.4.3）

IEEE 754 默认**向偶数舍入**（round half to even），不是"四舍五入"：

| 值 | 向偶数舍入结果 |
|----|----------------|
| 0.5 | 0（0 是偶数）|
| 1.5 | 2（2 是偶数）|
| 2.5 | 2（2 是偶数）|
| 3.5 | 4（4 是偶数）|

**工程意义**：大量数据累积时，向偶数舍入消除统计偏差；"四舍五入"会系统性偏高。

### 3. 浮点数不满足代数定律（§2.4.4）

浮点数**不满足结合律和分配律**，这与整数根本不同：

```c
// 结合律失败
(1e20 + (-1e20)) + 3.14  // = 3.14
1e20 + (-1e20 + 3.14)    // ≈ 0.0（3.14 被 1e20 吞掉）

// 分配律失败
1e20 * (1e20 - 1e20)     // = 0.0
1e20 * 1e20 - 1e20 * 1e20 // = NaN（∞ - ∞）
```

**工程意义**：编译器不能随意重排浮点运算（除非加 `-ffast-math`，但结果不精确）。

## 浮点陷阱汇总

### 陷阱1：大数吃小数（精度丢失）

```c
float x = 1e8f;
x = x + 1.0f;
printf("%f\n", x);  // 仍然是 100000000.000000，1.0 被吞掉
```

float 约 7 位有效数字，`1e8` 和 `1e8 + 1` 在 float 里是同一个数。

### 陷阱2：比较相等无意义

```c
double a = 0.1 + 0.2;
if (a == 0.3) { ... }  // 几乎永远 false，0.1/0.2/0.3 均无法精确表示
```

**正确做法**：`fabs(a - 0.3) < 1e-9`（根据精度需求选 epsilon）。

### 陷阱3：整数转浮点丢精度

```c
int x = 16777217;        // 2^24 + 1
float f = (float)x;
printf("%d\n", (int)f);  // 输出 16777216，丢了最后 1 位
```

float 尾数 23 位，能精确表示的整数范围是 $[-2^{24}, 2^{24}]$（即 ±16777216）。
`int` → `float` **不是无损转换**，超出范围的整数会被舍入。

### 陷阱4：NaN 的比较行为

```c
double nan = 0.0 / 0.0;
nan == nan   // false！NaN 不等于任何值，包括自身
nan != nan   // true
nan < 1.0    // false
nan > 1.0    // false
```

检测 NaN 必须用 `isnan(x)`，不能用 `==`。

### 陷阱5：有符号整数与浮点混合转换

```c
int x = -1;
float f = x;      // 正常：-1.0
unsigned u = x;   // 先隐式转 unsigned（变成 UINT_MAX），再转 float ≈ 4e9
```

### 陷阱6：浮点转整数截断溢出是 UB

| 转换 | 行为 |
|------|------|
| `int` → `float` | 可能舍入（尾数只有 23 位）|
| `int`/`float` → `double` | 精确（double 足够宽）|
| `double` → `float` | 可能溢出成 ±∞ 或舍入 |
| `float`/`double` → `int` | 向零截断；溢出是**未定义行为** |

```c
(int)(1e18)   // UB，不是截到 INT_MAX
(int)(NaN)    // UB
```

### 陷阱7：运算顺序影响精度

```c
// 计算调和级数 1 + 1/2 + 1/3 + ... + 1/N
// 从大到小加（先加大数）：精度差，小数被大数吞掉
// 从小到大加（先加小数）：精度好，同量级的数相加
```

---

## 易错点

| 易错点 | 原因 |
|--------|------|
| `0.1 + 0.2 != 0.3` | 三者在二进制下均无法精确表示 |
| `float` 精确表示整数的上限是 $2^{24}$ | 尾数只有 23 位 |
| `NaN != NaN` | IEEE 754 规定，NaN 与任何值比较结果均为 false |
| 浮点转 int 溢出是 UB | 不是截断到 INT_MAX，编译器可做任意处理 |
| 浮点加法不满足结合律 | 导致并行求和结果不确定 |

---

## 三句话总结

1. 浮点数是实数的近似，IEEE 754 用 1 位符号 + 阶码 + 尾数编码，规格化数隐含前导 1
2. 浮点运算不满足结合律，不能用 `==` 比较，NaN 不等于自身
3. 整数转 float 超过 $2^{24}$ 会丢精度，浮点转 int 溢出是 UB

## 工程关联

- 金融计算不能用浮点（用整数分/定点数）
- 并行求和结果因为结合律不同而不同是正常现象，不是 bug
- `isnan()`/`isinf()` 是处理特殊值的标准方式，不能用 `==`
- `-ffast-math` 允许编译器重排浮点运算，提升性能但牺牲精度，要谨慎使用
